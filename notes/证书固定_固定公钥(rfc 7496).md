# 证书固定_固定公钥(rfc 7496)

## 证书固定解决的问题

防止中间人攻击

## 证书固定对应的检查时机

在证书链通过验证以后执行证书固定检查

## RFC 7496中关于"证书固定检查通过"的定义

证书链如果要通过“证书固定检查”，需要满足如下条件

### 前置条件

这条证书链 **必须** 通过rfc 5280的验证(只满足条件1是通过不了的！)。

### 条件1

通过rfc 5280验证的证书链上证书的公钥hash构成的集合S;本地证书固定相关公钥构成集合P(Known Pins);

$$S \cap P \neq \emptyset$$

### 条件2

存在至少一个pin(backup pin)，它满足属于S或者P但是不属于它们的交集：

$$\text{pin} \in P \Delta S$$


### TTL（Time to Live）过期机制

RFC 规定固定信息必须带有过期时间。一旦超过该时间，该固定关系在客户端自动解除，退回到普通的 CA 信任链验证。

### 通过证书链子叶节点对应域名确定

大多数的实现都是通过“域名确定”，例如在android中:

**network_security_config.xml：**
```xml
<domain-config>
    <domain includeSubdomains="true">aaa.com</domain>
    <!--aaa-->
    <pin-set>
        <pin digest="SHA-256">替换为_aaa_第一个_PIN_码_BASE64================</pin>
        <pin digest="SHA-256">替换为_aaa_第二个_PIN_码_BASE64================</pin>
    </pin-set>
</domain-config>

<domain-config>
<domain includeSubdomains="true">bbb.com</domain>
<!--bbb-->
<pin-set>
    <pin digest="SHA-256">替换为_bbb_第一个_PIN_码_BASE64================</pin>
    <pin digest="SHA-256">替换为_bbb_第二个_PIN_码_BASE64================</pin>
</pin-set>
</domain-config>
```
通过rfc 5280验证的证书链：
```text
// 例子1
trust anchor -> ca1证书(bbb.com) -> 网站证书(aaa.com)

// 例子2
trust anchor -> ca1证书(bbb.com) -> 网站证书(gg.aaa.com)
```

例子1和例子2对应的P集合都为aaa注释的pin-set。

## Backup Pinning

RFC 7469 规范（关于 HTTP 证书固定）在制定标准时，明确强制要求至少必须包含一个备份公钥（Backup Pinning）

### 典型配置方案

**方案一：标准配置（1 主 + 1 备）—— 最常见**

**Pin A (主)**： 当前线上生产环境证书的公钥。

**Pin B (备)**： 一把生成后从未在线上部署过的备份私钥对应的公钥。这把私钥通常以绝对离线的方式（如冷存储、U盘、物理保险箱，或云厂商的顶级加密硬件 HSM）妥善保存。

**适用场景**： 绝大多数中大型互联网 App。

**方案二：高可用配置（1 主 + 2 备）—— 银行/金融级常用**

许多对稳定性要求极高的金融或大型 App，会准备 2 个备份公钥，用来防范不同维度的风险：

**Pin A (主)**： 线上当前使用的公钥。

**Pin B (原厂备份)**： 应对证书过期/密钥泄露。用于在主密钥出问题时，直接用这把备用密钥向同一个 CA 机构申请新证书。

**Pin C (跨 CA 备份)**： 应对 CA 机构本身破产或被吊销信任。比如你的主证书是 Digicert 签发的，Pin C 的备份密钥可能会跑去向 Let's Encrypt 
或 GlobalSign 申请一张证书。这样即使 Digicert 整个根证书体系崩溃，App 也能通过 Pin C 切换到新的 CA 机构，不至于全线瘫痪。

### 常见证书类型的签发耗时

* 域名型证书（DV - Domain Validation）—— 🚀 几秒钟到几分钟
* 企业型证书（OV - Organization Validation）—— ⏳ 1 到 3 个工作日
* 增强型证书（EV - Extended Validation）—— 🐢 3 到 7 个工作日
