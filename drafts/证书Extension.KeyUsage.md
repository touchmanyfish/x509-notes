# 证书Extension.KeyUsage


## 作用

**控制证书中公钥以及这个公钥对应的私钥的密码学用途**。

根据 RFC 5280 规范，如果证书包含了该扩展且将其标记为 `critical`（在生产实践中通常必须如此），那么客户端（如浏览器、操作系统、网络通信网关）在验证证书时，一旦发现实际发生的密码学操作超出了该扩展定义的范围，**必须拒绝该证书并中断通信**。

---

## Key Usage 的数据结构

在 ASN.1（Abstract Syntax Notation One，抽象语法标记）语法中，`KeyUsage` 被定义为一个 **`BIT STRING`（比特串）**。这意味着它的每一位都对应一个固定的布尔开关，`1` 表示允许该密码学操作，`0` 表示禁止。

### 1. ASN.1 原始定义

```asn
KeyUsage ::= BIT STRING {
     digitalSignature        (0),
     nonRepudiation          (1), -- 近期 RFC 中也称为 contentCommitment
     keyEncipherment         (2),
     dataEncipherment        (3),
     keyAgreement            (4),
     keyCertSign             (5),
     cRLSign                 (6),
     encipherOnly            (7),
     decipherOnly            (8) }

```

### 2. 物理编码与字节序表现形式

在基于 DER（Distinguished Encoding Rules，可辨别编码规则）的十六进制物理编码中，该比特串采用大端序（Big-Endian）排列：

* **第 0 位 (`digitalSignature`)** 对应第一个字节的最高位（MSB，即最左侧的 Bit）。
* 由于整个扩展目前定义了 9 个有效比特位，它会跨越 2 个字节（16 个比特位）进行存储。
* **填充规则：** 第二个字节中，除了第 8 位 (`decipherOnly`) 作为该字节的最高位有效外，剩余的 7 个低位比特位（Bits 9-15）在编码时会被强制补 `0` 补齐到字节边界。
* **物理排列示意：**

字节 1: [Bit 0] [Bit 1] [Bit 2] [Bit 3] [Bit 4] [Bit 5] [Bit 6] [Bit 7]
字节 2: [Bit 8] [  0  ] [  0  ] [  0  ] [  0  ] [  0  ] [  0  ] [  0  ]

## 9 个比特位（Bits）的详细语义与场景

| 位索引 | 标识符 (Identifier) | 允许的密码学操作 (Detailed Cryptographic Operations) | 实际作用与物理场景 (Real-world Scenarios) |
| :---: | :--- | :--- | :--- |
| **0** | **digitalSignature** | **数字签名生成与验证（基础实体认证）**<br>允许使用与证书内公钥对应的私钥，对特定的数据摘要（Hash）进行非对称加密运算生成数字签名。接收方利用证书公钥对其进行解密并比对摘要。此项通常用于基础的身份认证、实体鉴权或数据完整性校验，**不包含**下述专门针对不可否认性（Bit 1）、证书签名（Bit 5）或 CRL 签名（Bit 6）的特殊操作。 | • **TLS 握手中的客户端认证（Client Auth）：** 在双向 mTLS 认证中，客户端在 `CertificateVerify` 阶段用私钥对过往所有握手消息的 Hash 进行签名，向服务端自证拥有私钥。<br>• **临时密钥交换防篡改：** TLS 握手使用 **DHE 或 ECDHE** 算法时，服务器使用私钥对临时的 DH 公钥参数进行签名，防止中间人篡改参数。<br>• **代码签名（Code Signing）**等应用层基础认证。 |
| **1** | **nonRepudiation**<br>*(contentCommitment)* | **抗抵赖 / 不可否认性签名**<br>允许使用私钥对代表特定意愿、承诺或资产转移的数据实体进行数字签名。该操作在密码学、法律和审计层面提供最高级别的强绑定证据。它能够确保存储的数据在签署后未被更改，并且由于私钥的唯一性，防止私钥持有者事后声称该行为非本人意愿或属于伪造。 | • **电子合同与法律文献签署：** 确保签署协议在法律框架下具备不可推翻性。<br>• **金融交易与电子发票：** 银行间大额转账凭证签名、发票防伪签名。<br>• **安全电子邮件（S/MIME）：** 发件人对邮件正文进行不可否认性签名。 |
| **2** | **keyEncipherment** | **公钥加密（专门用于密钥传输 / Key Transport）**<br>允许直接使用证书中的公钥去加密**另一个对称密钥（例如 AES 密钥或 TLS 预主密钥 Pre-Master Secret）**。加密后的密文通过公开通道传输，只有对应的私钥持有者能够将其解密并还原出对称密钥，从而建立起临时的对称加密通道。 | • **传统 RSA 密钥交换模式下的 TLS 握手：** 客户端生成 Pre-Master Secret，直接用服务器证书里的 RSA 公钥加密后发送给服务器（注意：TLS 1.3 已废除此模式）。<br>• **S/MIME 邮件加密：** 发送方生成临时的对称密钥加密邮件正文，然后用接收方的公钥加密该对称密钥并附在邮件头部。 |
| **3** | **dataEncipherment** | **公钥直接加密（专门用于原始数据 / Data Encryption）**<br>允许直接使用该公钥对**应用层原始数据或配置信息**进行非对称加密，而不是加密对称密钥。由于非对称加解密的数学计算开销极大、性能极低，且有最大加密长度限制，因此此操作在工程实践中极少使用，且严格禁止与 `keyEncipherment` 混用。 | • **超小体积敏感数据的直接非对称存储：** 例如在极高安全级别的密码机（HSM）中，直接用公钥加密保存一段核心 PIN 码、硬编码 Token 或极小的敏感配置块。 |
| **4** | **keyAgreement** | **密钥协商 / 密钥对齐 (Key Agreement)**<br>允许证书持有者与通信对端通过特定的密码学协议（如 **Diffie-Hellman (DH) 或 Elliptic Curve DH (ECDH)**）共同参与推导出一个共享的对称密钥。在这个过程中，公钥的作用是作为协商因子参与数学计算，而不是用来直接“加密”或“签名”任何外部数据。 | • **静态密钥交换 (Static DH/ECDH)：** 通信双方使用证书中固定的 DH 公钥与对方的临时或固定公钥通过数学公式（如 $g^{ab} \pmod p$）直接算出对称密钥。<br>• **专用安全隧道：** 某些电信级标准（如 IPsec 某些模式）或 VPN 专用基础设施中的对等实体身份与密钥协商。 |
| **5** | **keyCertSign** | **证书签名验证（CA 专属核心权限）**<br>**PKI 信任链传递的绝对基石。** 允许使用该公钥去验证**下级非对称数字证书（Certificate）上的数字签名**。在整个 PKI 体系中，这意味着该证书持有者是一个合法的、有权发证的认证中心（CA）。只有 CA 证书允许且必须开启这一位。 | • **根 CA (Root CA) 对二级子 CA (Subordinate CA) 证书的签名与发行。**<br>• **子 CA 对最终实体（如网站的 TLS/SSL 证书、代码签名证书）的签名与发行。** |
| **6** | **cRLSign** | **吊销列表（CRL）签名验证**<br>允许使用该公钥去验证**证书吊销列表（CRL，Certificate Revocation List）**或者吊销状态数据（如 OCSP 响应组件）上的数字签名。这属于 PKI 体系中的身份退出/作废管控权限。 | • **CA 机构定期发布并签署 CRL：** 声明哪些证书在有效期内已被提前吊销。<br>• **指定吊销发布点 (CRL Issuer)：** CA 将吊销职责委派给专门的独立吊销签名服务器，该服务器证书会单独开启这一位。 |
| **7** | **encipherOnly** | **仅用于加解密协商中的加密（配合 Bit 4 限制使用）**<br>该位**必须且只能在 `keyAgreement` (Bit 4) 同时为 1 时**才有意义。当进行密钥协商并推导出共享的对称密钥后，限制该对称密钥**只能用于加密数据**，绝对不能用于解密。 | • **单向单工数据收集网络：** 例如机密物联网传感节点或军事上报终端，它利用密钥协商建立通道后，物理上只允许向中心“单向发送加密数据”，防止节点被物理攻破后反向解密别人的流。 |
| **8** | **decipherOnly** | **仅用于加解密协商中的解密（配合 Bit 4 限制使用）**<br>该位**必须且只能在 `keyAgreement` (Bit 4) 同时为 1 时**才有意义。当进行密钥协商并推导出共享的对称密钥后，限制该对称密钥**只能用于解密数据**，绝对不能用于加密。 | • **单向单工数据接收网关：** 专门的中心数据接收隔离区（DMZ），利用协商密钥后，该节点在物理和算法限制下“只能解密接收外界数据”，无法向外发送合法的加密报文。 |

## 在TLS 1.2握手中使用举例

### 密钥交换算法选择rsa

server证书设置：
```text
// 用不到这个设置为0
digitalSignature = 0 

// (公钥加密其他密钥)client使用server证书中的公钥加密pre-master-secret
keyEncipherment = 1 
```

### 密钥交换算法使用DHE

server证书设置：

```text
// (签名生成)server用自己证书中公钥对应的私钥生成签名，放入ServerKeyExhange中发给client
// (认证)client需要使用server证书中的公钥来验证签名
digitalSignature = 1 

// 用不到这个，所以设置为0
keyEncipherment = 0 
```

### server要求双向验证

client证书设置:
```text
// (签名生成)client用自己证书中公钥对应的私钥生成一个签名，放入CertificateVerify中发给server
// (认证)server会用client证书中的公钥来验证CertificateVerify中包含的签名
digitalSignature = 1
// 用不到这个，所以设置为0
keyEncipherment = 0
```

## 在 CA 与 PKI 体系中的核心作用

### 1. 明确角色边界与特权隔离（CA vs Leaf）
在 PKI 治理体系中，最重要的安全防线就是**防止叶子证书（终端用户/最终实体证书）冒充或越权充当 CA**。
* **CA 证书：** 必须在 `KeyUsage` 中显式设置 `keyCertSign = 1`。
* **终端实体证书（如你开发中常见的 HTTPS 网站证书）：** 其 `keyCertSign` 必须为 `0`。

> **防攻击安全边界：** 假设黑客合法购买了一个正规的 HTTPS 网站证书，并尝试用该证书对应的私钥去伪造签署另一个高价值域名的伪造证书。当客户端（如 Chrome 浏览器或 Android 的系统组件）在建立 TLS 连接验证这个伪造证书链时，会追溯到黑客的那个原版网站证书，并检查其 `KeyUsage` 扩展。一旦发现其 `keyCertSign` 位为 `0`（无权签发证书），客户端会立即判定该证书链断裂，直接拦截并报证书不可信错误。

### 2. 职责分离原则（Separation of Duties）
高安全等级的公有或私有 CA 为了降低系统性风险，不会将所有的核心权力集中在同一个证书/私钥主体下。通过 `KeyUsage` 可以实现精细化的权限分拆：
* **Issuing CA（证书签发服务器）：** 设置 `keyCertSign = 1`，但可以设置 `cRLSign = 0`。它只负责源源不断地签署和发行新证书。
* **CRL Signer（吊销基础设施）：** 设置 `keyCertSign = 0`，但设置 `cRLSign = 1`。这个私钥被限制在专门的吊销列表生成集群中，即使该私钥不幸泄露，黑客也**无法利用它来伪造签署任何一个合法的数字证书**，从而将损失控制在单一维度内。

## 补充：Key Usage 与 Extended Key Usage (EKU) 的区别

这两者在名称上高度相似，但在 X.509 证书的解析、校验和合规体系中，它们属于**技术原语**与**业务场景**两个完全不同、互相交织的补充维度。

| 维度 | Key Usage (KU) | Extended Key Usage (EKU) |
| :--- | :--- | :--- |
| **核心关注点** | **密码学技术原语（How）**<br>关注的是“这个公钥在数学和物理层面，能做什么**纯密码学操作**”（如：能不能做数字签名、能不能做非对称加密、能不能推导密钥）。 | **业务应用场景（Where / What Role）**<br>关注的是“这个证书被允许用在**什么具体的上层业务协议、框架或角色中**”（如：Web 服务器、代码签名、客户端认证、时间戳）。 |
| **RFC 定义依据** | RFC 5280 内部明确定义，结构固定。 | RFC 5280 定义通用框架，具体业务的 OID 由各个专门的业务 RFC（如 TLS、S/MIME 规范）独立扩充定义。 |
| **数据表达结构** | **`BIT STRING`（比特位映射）**<br>固定且紧凑的 9 个比特位。 | **`ASN.1 OBJECT IDENTIFIER (OID)` 数组**<br>可无限扩展的 OID 字符串数组。<br>例如：`1.3.6.1.5.5.7.3.1` (表示 Server Auth) |
| **典型实例** | 无论证书是给 Web 服务器用、给 Android 开发者进行代码签名用，还是给邮件用，只要上层算法是 RSA 且需要做密钥传输，其 KU 均勾选 `keyEncipherment`。 | • **Web 服务器证书：** 必须包含 `Server Auth OID`<br>• **开发者软件代码签名证书：** 必须包含 `Code Signing OID`<br>• **客户端身份证书：** 必须包含 `Client Auth OID` |

### 联合双重校验逻辑（双重锁机制）
在现代网络安全验证（如 Android 系统的网络安全配置、操作系统的根证书校验、各类 Web 浏览器）中，客户端在检查证书时会采取 **“KU 与 EKU 必须同时交集满足”** 的死理：

* **案例：一个合法的 HTTPS 网站服务器证书（TLS Server Certificate）**
  1. **底层物理锁 (Key Usage)：** 必须满足 `digitalSignature = 1`（如果使用 ECDHE/DHE 算法协商） 并且/或者 `keyEncipherment = 1`（如果使用传统 RSA 密钥交换模式）。
  2. **上层应用锁 (Extended Key Usage)：** 必须在 OID 列表中包含 `id-kp-serverAuth` (`1.3.6.1.5.5.7.3.1`)。
  3. **安全防护结果：** 就算黑客完全控制了该网站的证书和私钥，并试图将它塞进一个开发者代码签名工具里去对一个恶意的 `.apk` 或 `.exe` 驱动进行代码签名，签名工具在校验时会发现该证书的 EKU 中缺少 `id-kp-codeSigning` (`1.3.6.1.5.5.7.3.3`) 的 OID。由于扩展无法匹配上层业务场景，校验会立刻失败。这就是通过 KU 与 EKU 的双重锁死，保障了整个数字证书生态的各司其职。
